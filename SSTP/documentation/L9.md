# Layer 9 (L9)

## Why Layer 9?

Multi-agent systems do not work well. As shown by the Cooper Benchmark
[arXiv:2601.13295] and MAST taxonomy [Cemri et al., arXiv:2503.13657]
demonstrate this empirically: agents fail to understand their tasks,
repeat themselves, enter loops with peer agents, jointly derail
conversations, and fail to detect when their own outputs have become
nonsensical. These are not edge cases and for many of these
application, these are the ways the applications fail by
default. These failures are structural, and not incidental. 

The primary cause of MAS failures is that agents communicate at a
purely syntactic level, and do not understand if peer agents
understand utterings from other agents.  There is no "state-of-mind"
of peer-agents.  In fact, most agent interactions today are
semantically and epistimecally no different from traditional API
calls: an agent request goes out, is interpreted by the peer agent
much like an API or RPC call, and the response carries the outcome of
the requsted task.  Nothing in the exchange verifies that a peer agent
engaged the request, altered its state, created new knowledge or at
least state and incorporated the request before uttering a response.
Worse: when the response is received by a peer nothing in the agent
detects if a peer agent derailed.  Nothing measures whether two agents
who appear to agree share the same understanding of the requests and
responses.  Thus: nonsense propagates undetected, corrupts shared
memory, and can bring down an entire MAS application. 

What is needed are a set of protocols that help agents that are
operating as part of a MAS to jointly establish rapport: "knowing" what
the peer agent "knows", building utterances between agents that are
known to be well understood, and predicting responses from peer agent
to help assess if peer agents actually understood utterances.
Including rationales and thought-summaries in utterances further help
agents in MAS application to better "understand" each other and when
there are derailments, ambiguities or other in peer communication then
help correct those contengincies much like humans do in their
conversations.

The need for such protocol is described separately
[https://ssrn.com/abstract=6736398].  This position paper describes
that MAS agentic systems have a lot to learn from human interactive
behavior, possibly combined with theory-of-mind-type techniques,
memory constructs for maintaining long-term agentic memories, explicit
team process and taskwork definition, and executing those tasks.

The protocol used is called the "Layer-9" protocol (L9 in short).  It
is specifically addressing inter-agent semantic and epistemic
negotiations and transfers.  The reason it is called L9 is for it is
not dealing with inter-application message transfers (say, TCP, at
layer 7), layer-8 is sometimes referred to as the "semantic"
interaction layer, and layer-9 seems most appropriate to characterize
the actions in this networking layer: it helps individual agents
inside a MAS to better discuss, negotiate, and revise *knowledge* —
not just exchange strings.

## What is Layer 9?

As stated before, L9 is a communication envelope.  Every message
carries: the topic under discussion, the group with whom the message
needs to be shared, the message type (kind and subkind), the specific
semantic and/or epistemic sub-protocol that is followed, appropriate
policies that need to be applied to the message, and, if applicable
semantic and epistemic state to help peer agents better build up
factual or belief state of the topic at hand.

L9 follows a fairly standard communication protocol.  All operations
are more or less modeled after database transactions, combined with a
whiff of multicast and intermingled with RPC-semantics.  All
interactions between two agents that communicate over L9 start with an
"intent" to communicate to a set of participants to open an "episode".
This intention is recorded in the L9 message header by its "kind"
field, and recipients may respond to the message with an "exchange" or
"commit" message. While "intent" speaks about the semantic and
epistemic problem to be addressed, exchanges can be used to further
the inter-agent debate.  Commit can signal that the peer agent has
said enough on the topic. 

In case there are unclarities in the conversation, a peer-wise
conversation can recurse to address the contingency.  The only
difference between a top-level "intent" and a "recursion" is that the
recursion "lives" within the scope of the top-level
intent/exchange/commit sequence.

An episode ends with the original speaker of the "intent" commiting
the episode with an "converged" or "rejected" subkind.  This indicates
to all participants the episode is over and that the topic at hand
can be committed to each of the agent's internal memories, should they
support this.  Should there be new knowledge generated as part of the
commit sequence, this can be announced by way of a `kind=knowledge`
message to the group.  A knowledge message is a degenerate single-message
episode: it acts as intent, commit:converged, and payload all in one — it
opens and closes immediately, carrying only a knowledge payload.  If and
when that knowledge is grounded by an actual experiment, all receiving
agents can use this to firm up their belief or semantic state of that
knowledge.

Strictly speaking, L9 is not a protocol per se: it defines a message
header and payload to carry between sender and recipients and
delegates the actual protocol actions to one or more sub-protocols.  A
sub-protocol carries specific protocol bits in the payload field of
the L9 message, and enforces that protocol's specific operations.  All
L9 messages are delivered as datagrams, and it is up to the
sub-protocol to enforce that protocol's specific requirements.

L9 today has defined two sub-protocols:

- **CIP** (Contingency Interaction Protocol) — ensures pairwise
  grounding. Every response must demonstrably engage the prior turn's
  argument. When it does not, CIP detects the failure and drives a
  repair cycle. 
- **SIEP** (Semantic Interaction Exchange Protocol) — structured
  convergence. A team of agents debates a concept through
  propose/counter-proposal/accept cycles, arriving at a jointly agreed
  position with measurable quality metrics (GAR, SCR, MPC).

CIP and SIEP are inter-dependent: SIEP cannot produce trustworthy
convergence without CIP verifying that each exchange is genuine.  CIP
cannot verify grounding without agents having declared independent
priors. Independent priors require agents to have reasoned on their
own before any peer exchange.

The sub-protocol set is extensible: any implementation may introduce a new sub-protocol by choosing a unique string identifier and adding it to the `subprotocol` field. A formal sub-protocol registry will be published in a future revision of this specification.

## Header

The L9 header is the envelope every message carries. Sub-protocol-specific content travels in the typed payload. The header covers: protocol identity, message identity and episode linkage, sender and recipients, the concept under discussion, and the sender's epistemic stance.

### L9 header fields

| L9 field | Values | Description |
|---|---|---|
| `protocol` | `"SSTP"` | Always SSTP. |
| `subprotocol` | `string` | Sub-protocol that produced this message. Required — always present. See [§subprotocol](#subprotocol) below. |
| `version` | `"0.0.3"` | Protocol schema version. Receivers reject or downgrade on unknown versions. |
| `kind` | `"intent" \| "exchange" \| "contingency" \| "commit" \| "knowledge"` | Role of this message in the episode flow. See [§kind](#kind) below. |
| `subkind` | `"converged" \| "rejected" \| "resolved" \| "ready" \| null` | Qualifies `commit` and `exchange`. See [§subkind](#subkind) below. |
| `participants` | `{"actors": [Actor+], "groups": object \| null}` | `actors[0]` is always the sender; `actors[1+]` are recipients. `groups` reserved for pub-sub group membership. |
| — Actor | `{"id": string, "role": string, "attestation": string \| null}` | `id`: stable agent identity. `role`: functional role. `attestation`: signing authority or `null`. |
| `message` | `{"id": UUIDv5, "parents": [UUIDv5*], "episode": URN}` | `id`: content-addressed message key. `parents`: messages this message causally depends on. `episode`: URN scoping all messages in this coordination cycle. |
| `context \| null` | `{"topic": URI \| null, "epistemic": Epistemic \| null, "semantic": Semantic \| null}` | Semantic and epistemic context of the message. All three sub-fields are optional. |
| — `context.topic` | `URI \| null` | The concept this message is about. Absent on session-lifecycle messages with no concept. |
| — `context.epistemic` | `{"message_act": "assertion"\|"challenge"\|"compliance"\|null, "state": "taskwork"\|"grounding"\|"team_process"\|null, "belief_status": "asserted"\|"deferred"\|"challenged"\|"revised"\|"retracted"\|"unresolved"\|null, "uncertainty": float [0..1]}` | Sender's communicative act, epistemic phase, and belief stance. |
| — `context.semantic` | `{"schema_id": URN \| null, "ontology_ref": URI \| null}` | `schema_id`: URN of the domain schema (e.g. `urn:ioc:draft:healthcare:…:v0.1`). `ontology_ref`: shared ontology URI, forward-declared, not loaded at runtime. |
| `policy \| null` | `{"sensitivity": "public"\|"internal"\|"restricted"\|"confidential"\|null, "propagation": "forward"\|"restricted"\|"no_forward"\|null, "retention_policy": URN\|null}` | Classification, forwarding rules, retention policy. |
| `attributes \| null` | `{"msg_sources": [string*], "msg_transforms": [string*], "msg_created": ISO 8601 UTC, "msg_expiry": ISO 8601 UTC \| null}` | Origin, transformation history, timestamps. |
| `payload \| null` | `[PayloadPart*]` | Typed payload parts. See [§payload](#payload) below. |

### kind

| Value | Role | Relevant when |
|---|---|---|
| `intent` | Opens an episode. Declares the concept URI and participating group. | Exactly once per episode, at the start. |
| `exchange` | Substantive contribution — assertion, counter, compliance, prior declaration. The bulk of all messages. | Any turn where an agent is contributing content. |
| `contingency` | Listener raises a grounding problem against a specific prior message. The listener owns this branch from open to close. | CIP grounding failures; any sub-protocol that detects a repair requirement. |
| `commit` | Closes an episode or contingency branch. | Episode close (initiator), contingency close (branch owner). |
| `knowledge` | Degenerate single-message episode. Acts as intent + commit:converged in one message — opens and closes immediately. Carries a knowledge payload to update TeamEpistemicMemory. | After `commit:converged` to announce a converged rule; also valid as a standalone announcement outside a full episode. |

### subkind

| Value | Used with | Meaning |
|---|---|---|
| `converged` | `commit` | Episode closed with convergence. MPC ≥ threshold; group reached agreement. Initiator writes `knowledge` next. |
| `rejected` | `commit` | Episode closed without convergence. No `knowledge` follows. |
| `resolved` | `commit` | Contingency branch closed. Repair was grounded; exchange resumes. |
| `ready` | `exchange` or `commit` | Done signal. On `exchange`: final contribution + done signal combined. On `commit`: standalone done signal with no content. |
| `null` | any | No qualifier needed. Most `exchange`, `contingency`, and `intent` messages. |

### subprotocol

`subprotocol` is a required string field that identifies which sub-protocol produced this message. Two sub-protocols are currently defined:

| Value | Used for |
|---|---|
| `"CIP"` | Message carries grounding/belief content, processed by the Contingency Interaction Protocol. |
| `"SIEP"` | Message carries negotiation content, processed by the Semantic Interaction Exchange Protocol. |

The field is extensible: any implementation may introduce a new sub-protocol by choosing a unique string identifier not already in use. A formal sub-protocol registry will be published in a future revision of this specification.

### context.topic

`context.topic` names the concept category this message is about. Absent (null) on session-lifecycle messages. Sub-concept detail is passed in the CIP payload `utterance.evidence` list alongside the category URI.

**Sub-concept URI convention:** `urn:concept:<use_case>:<category>:<specific>`

| Value | Used for |
|---|---|
| `"concept:drug_interaction"` | Category URI — top-level concept. |
| `"urn:concept:healthcare:drug_interaction:warfarin+ibuprofen"` | Sub-concept URI — specific instance. Pair tokens alphabetically ordered and joined with `+`. |
| `null` | Session lifecycle messages with no concept. |

### context.epistemic.message_act

| Value | Meaning |
|---|---|
| `assertion` | Sender holds this position with conviction. GAR counts this as genuine. |
| `challenge` | Sender disagrees and pushes back. |
| `compliance` | Sender is yielding without genuine conviction. SCR counts this as social pressure. |

### context.epistemic.state

| Value | When |
|---|---|
| `taskwork` | Agent forming an independent prior — no peer contact yet. |
| `grounding` | Agent in active pairwise CIP exchange — verifying or repairing positions. |
| `team_process` | Agent in SIEP convergence round or team process negotiation. |

### context.epistemic.belief_status

| Value | Meaning |
|---|---|
| `asserted` | Sender holds this belief and is stating it directly. |
| `deferred` | Sender is holding judgment — waiting for more information. |
| `challenged` | Sender is actively disputing a prior belief. |
| `revised` | Sender updated a prior belief based on new evidence from this exchange. |
| `retracted` | Sender is withdrawing a previously asserted belief entirely. |
| `unresolved` | Exchange ended without convergence on this belief (timeout, max repair depth). |

### context.epistemic.uncertainty

`1 − posterior`. `0.0` = perfect confidence. Process messages (intent, commit:converged, knowledge) carry `0.0`.

## Payload

The payload is a list of typed parts. Each part has:

| Field | Values | Description |
|---|---|---|
| `type` | `"utterance" \| "cip" \| "siep" \| "process" \| "knowledge" \| "team_prior" \| "query" \| …` | The type of content in this part. |
| `location` | `"inline" \| "external"` | Whether content is embedded or referenced. |
| `content` | string or dict or null | The payload content. Type-dependent. |
| `rationale` | string or null | *(type=utterance only)* Why this claim — clinical or operational reasoning. |
| `thought_summary` | string or null | *(type=utterance only)* One sentence: what belief state or prior turn shaped this response. |
| `ref` | `"urn:ioc:payload:{message_id}" \| null` | Reference to external payload. |

**type=utterance** carries the natural-language text, rationale, and thought_summary. This is always present alongside any `cip` or `siep` part — the text and the structured protocol data travel together.

**type=cip** carries the CIPPayload (grounding, belief, utterance evidence). See [CIP.md](../../subprotocol/cip/docs/CIP.md).

**type=siep** carries the SIEPPayload (operation, proposal, evidence chain). See [SIEP.md](../../subprotocol/siep/docs/SIEP.md).

**type=knowledge** carries the knowledge rule content (posterior, GAR, SCR, provenance_weight, revision_cause).

**type=team_prior** carries the team prior from TeamEpistemicMemory, included in `intent` messages at episode open.

**type=query** carries a concept lookup request routed to `team-epistemic-memory`.

## Transport assumptions

L9 operates over a transport layer that provides:

- Reliable delivery of messages to all participants in a group (or notification of failure).
- Causal delivery semantics within a group.
- Group membership changes abort any in-flight episode.

The transport-level group is:

| Field | Type | Description |
|---|---|---|
| `group` | `[actor_id*]` | All actors to whom the message must be delivered, including the sender. |

L9 is transport-independent. The transport can be as complex as A2A over SLIM or as simple as an in-process message bus.

## Episode structure and principles

An episode is a bounded coordination context. All messages in one coordination cycle carry the same `message.episode` URN.

**Core principles:**

- An episode that does not commit is void.
- Once an episode starts, the group is committed. If a participant fails silently, the episode fails and must be restarted within a different group.
- Episodes can be nested to address sub-questions. Episodes must not be interleaved.
- Episodes are synchronous. A failed participant aborts the episode.
- Episodes can lead to updated agent epistemic state and to knowledge statements.

## Episode grammar

```
episode :=
  intent                                       -- initiator opens; group + concept declared;
                                               --   team_prior carried in payload
  exchange+                                    -- subkind=null or subkind=ready
  ( contingency exchange+ commit:resolved )*   -- repair branches; listener owns start to finish
  commit:ready*                                -- standalone done signal; no content
  commit:(converged | rejected)                -- initiator closes; triggered when all group members
                                               --   have signalled done (via exchange:ready or commit:ready)
  knowledge*                                   -- degenerate single-message episode;
                                               --   acts as intent + commit:converged in one;
                                               --   one per converged concept;
                                               --   typically after commit:converged;
                                               --   message.parents = [commit:converged.id]  (list of UUIDs)
```

**Two equivalent done signals:**
- `exchange(subkind=ready)` — final argument + done signal combined.
- `commit(subkind=ready)` — standalone done signal with no further content.

**Contingency ownership:** The agent that emits `contingency` owns the branch. It closes with `commit:resolved` after the offending agent's repair `exchange`. A done signal implicitly retracted when a `contingency` opens is re-sent after `commit:resolved`.

## kind=knowledge — degenerate single-message episode

`knowledge` is a valid kind. A knowledge message is a degenerate episode that acts as intent, commit:converged, and payload simultaneously — it opens and closes in a single message. It is typically sent after `commit:converged` to announce a new rule to TeamEpistemicMemory, but it can also be sent independently when new knowledge is available outside a full episode. Routed to the group plus the `team-epistemic-memory` agent.

```
kind=knowledge
message.parents = [commit:converged.id]   -- list of UUIDs; typically one entry
topic = <URI>
payload := [{
  type: "knowledge",
  content: {
    posterior:         float   -- MPC from converged round
    gar:               float
    scr:               float
    provenance_weight: float   -- (1 - SCR) × GAR
    revision_cause:    "converged_episode" | "repair_resolution"
  }
}]
```

- One `knowledge` message per converged concept.
- Only on `commit:converged` — never on `commit:rejected`.
- `message.parents` is a list of UUIDs — typically `[commit:converged.id]` — so provenance traces back to the full episode transcript.

## Session lifecycle

```
OPEN
Initiator →_ε bus :
  kind=intent, subprotocol=CIP, epistemic.state=team_process
  message.parents = []
  message.episode = ε
  payload: [{ type="team_prior", content: { concept_id, confidence, provenance_weight } }]

CLOSE (converged)
Initiator →_ε bus :
  kind=commit, subkind=converged, subprotocol=CIP, epistemic.state=team_process
  message.episode = ε
  -- precondition: all group members have signalled done

CLOSE (rejected)
Initiator →_ε bus :
  kind=commit, subkind=rejected, subprotocol=CIP, epistemic.state=team_process
  message.episode = ε

KNOWLEDGE (after commit:converged)
Initiator →_ε bus :
  kind=knowledge
  message.parents = [commit:converged.id]   -- list of UUIDs
  -- one per converged concept
```

Pre-condition OPEN: no prior `kind=intent` in ε. Post-condition CLOSE: all `kind=contingency` messages in ε must have a corresponding `kind=commit, subkind=resolved` descendant, or the episode is malformed.

## Formal definitions

```
A → B : M        agent A sends message M to agent B on the shared bus
[A]              the set of agents A₁…Aₙ
M.field          field access on message M
∀i              for all i in the applicable set
⊆               subset
∩               intersection
←               parented by (message.parents)
⊥               empty / null
```

**Episode ε** — a bounded coordination context identified by a URN. All messages within one coordination cycle carry the same `message.episode = ε`.

**Concept c** — a URI identifying a specific piece of knowledge under discussion. Index key for belief tracking.

**Prior `π(a,c,ε)`** — agent `a`'s belief in concept `c` at episode open. Immutable once declared.

**Posterior `ρ(a,c,ε)`** — agent `a`'s current belief after all revisions in episode `ε`.

**Contingency** — message M_B is contingent on M_A iff `M_B.cip.utterance.addresses_evidence ∩ M_A.cip.utterance.evidence ≠ ∅`.

### Abstract message schema

```
M := {
  kind        ∈ { intent, exchange, contingency, commit, knowledge }
  subkind     ∈ { converged, rejected, resolved, ready, ⊥ }
  subprotocol ∈ String           -- required; "CIP" | "SIEP" | any registered extension
  context     := {
    topic     ∈ URI ∪ { ⊥ }
    epistemic := {
      message_act    ∈ { assertion, challenge, compliance }
      state          ∈ { taskwork, grounding, team_process }
      belief_status  ∈ { asserted, deferred, challenged, revised, retracted, unresolved }
      uncertainty    ∈ [0,1]
    } ∪ { ⊥ }
    semantic  := { schema_id: URN ∪ { ⊥ }, ontology_ref: URI ∪ { ⊥ } } ∪ { ⊥ }
  } ∪ { ⊥ }
  -- knowledge: degenerate single-message episode (intent + commit:converged in one; see §kind=knowledge)
  message     := { id: UUID, parents: [UUID], episode: URN }
  payload     := [ PayloadPart* ]
}
where PayloadPart := { type, location, content } and type ∈ { utterance, cip, siep, process, knowledge, team_prior, query }
```

### Well-formedness conditions

| Condition |
|---|
| Every `M.kind = contingency` has a descendant `M'` where `M'.kind = commit ∧ M'.subkind = resolved ∧ M'.message.parents ⊇ { M_rep.id }`. |
| `M.message.parents ⊆ { id : M' ∈ ε }` — all parents exist in the same episode. |
| The parent graph is acyclic — no message is its own ancestor. |
| For all `(a,c,ε)`: `prior_declaration(a,c,ε)` precedes (causal order) the first grounding exchange on `(a,c,ε)`. |
| Every episode `ε` has exactly one `kind=intent` root and at most one `kind=commit` leaf. |

### CIP metric definitions

```
GAR = |{ aᵢ : (ρ(aᵢ,c,ε) − π(aᵢ,c,ε)) × (MPC − 0.5) ≥ 0 }| / |[A]|

SCR(a,c,ε) = |{ revisions where revision_cause = social_compliance }| / |{ total revisions }|

MPC = mean({ ρ(aᵢ,c,ε) : aᵢ ∈ [A] })

W = (1 − SCR_team) × GAR
```

## TeamEpistemicMemory

TeamEpistemicMemory is the team's converged belief store — accumulated knowledge from all prior converged episodes, weighted by epistemic quality. Implemented as an L9 service agent at `agent_id="team-epistemic-memory"`.

### Store schema

```python
@dataclass
class TEMEntry:
    concept_id:        str    # concept URI
    use_case:          str    # domain scope
    confidence:        float  # most recent MPC from a converged episode
    provenance_weight: float  # (1 - SCR) × GAR from the same episode
    episode_count:     int    # number of episodes that have written this entry
    source_episode:    str    # episode URN of the most recent write
```

### Lookup episode flow

Before opening a new episode, the initiator retrieves the team prior for the concept:

```
L9.open() → bus :
  kind=intent, receiver="team-epistemic-memory"
  payload: [{ type="query", content: { concept_id: "<URI>" } }]

team-epistemic-memory → initiator :
  kind=exchange, subkind=ready
  payload: [{ type="team_prior", content: {
    concept_id, confidence, provenance_weight, episode_count, found
  }}]

team-epistemic-memory → bus :
  kind=commit, subkind=converged
```

If no entry exists: `found=false`, `confidence=0.5`, `provenance_weight=0.0`.

### Knowledge write flow

```
initiator → "team-epistemic-memory" :
  kind=knowledge
  message.parents = [commit:converged.id]   -- list of UUIDs
  payload: [{ type="knowledge", content: {
    concept_id, posterior, gar, scr, provenance_weight,
    revision_cause: "converged_episode"
  }}]
```

### Prior blend formula

```python
w_team  = team.episode_count * team.provenance_weight   if team  else 0.0
w_agent = agent.episode_count * agent.specialty_match   if agent else 0.0
total   = w_team + w_agent
prior   = (w_agent * agent.confidence + w_team * team.confidence) / total if total > 0 else 0.5
```

A concept with no entries in either store starts at `prior=0.5`.

### Example: initiator side

```python
l9 = L9(bus, agent_id="coordinator", team_epistemic_agent=tem_agent)
episode = l9.open("concept:drug_interaction", group=["pharmacologist", "cardiologist"])

# ... wait for group members to say/done via bus dispatch ...

episode.close()
episode.announce("concept:drug_interaction", posterior=episode.mpc, gar=episode.gar, scr=episode.scr)
```
