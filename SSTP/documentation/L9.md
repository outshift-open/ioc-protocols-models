
### Why Layer-9?

Multi-agent systems do not work well. The Cooper Benchmark and the MAST taxonomy paper by Cemri et al. demonstrate this empirically: agents fail to understand their tasks, repeat themselves, enter loops with peer agents, jointly derail conversations, and fail to detect when their own outputs have become nonsensical. These are not edge cases — they are the default. The consequence is that MAS applications today are largely unreliable at scale, and the failures are structural, not incidental.
The root cause is that agents communicate at a purely syntactic level.

Agent interactions today are semantically no different from traditional API calls: a request goes out, a response comes back. Nothing in the exchange verifies that the response engaged the request. Nothing detects when an agent has derailed. Nothing measures whether two agents who appear to agree share the same understanding. Nonsense propagates undetected, corrupts shared memory, and can bring down an entire application.

This is the problem L9, IE, and SNP exist to solve: L9 is an essential function to uplift traditional agents as these are operating in a multi-agent system (MAS) from their traditional API-style interactions, into one where these agents are discussion, negotiating and dismissing epistemic state, or simply: knowledge.

L9 is about enabling agents in a MAS to debate and negotiate their ground truth, and be moved into different positions on epistemics (= knowledge) given multi-agent social pressure or general agreements.  To get to these points, agents discuss “concepts”, the knowledge at hand, and use theory-of-mind-based convergence and grounding concepts to settle jointly on those concepts.   These discussions are called “episodes”.

### What is Layer 9?

To make agents manage and debate knowledge, a *L9 framework* is made available for agents to engage with that knowledge maintained by itself in combination with other L9-capable agents.  The framework is a set of protocols, support functions, libraries and at some point skills to uplift existing agents into epistemic distributed agents (= a set of agents that debate knowledge).  The framework is grounded in three observations about what needs to work in a MAS application.

- Agents must independently form calibrated beliefs before any exchange with another L9 capable agent.  An agent that has not done its own reasoning has nothing genuine to contribute.  Without those independent *priors*, convergence about knowledge is meaningless — agents will simply agree with whoever spoke last.
- When agents exchange their positions, each response must demonstrably engage the prior turn’s argument. A response that does not, is not a response — it is a parallel assertion. The system must detect this and trigger repair.  This is what the Interaction Engine does.
- The team (or swarm) of agents must converge on a shared position through structured argument, and the quality of that convergence must be measurable and recorded.  A nominal agreement that nobody believes is worse than no agreement, because it masquerades as settled knowledge.

L9 is the communication envelope that makes all three problems tractable in MAS systems.  It is a transport protocol to carry, on every message, the concept under discussion, the sending agent's belief state about that concept, the phase of reasoning much like humans reason, and a typed sub-protocol, with a sub-protocol being a specific epistemic protocol under L9 to address a specific knowledge aspect, payload.

L9 today has two sub-protocols are layered and that are inter-dependent.   The semantic negotiation protocol (SNP) is a convergence protocol: it establishes a reasoned converged outcome given a task that needs to be executed.  For this in a group of agent, a reasoned debate is exercised where participants jointly come to a specific resolution for the task at hand.  For instance, in a hospital case with a set of agentic physicians, the physicians can reason with each other given a specific ailment of a patient.  Each physician needs to argue from its speciality, while acknowledging the other physician’s point of view.

A separate sub-protocol is the iteraction engine (IE).  Named after the human equivalent per Levinson’s work, this agentic version makes certain that ambiguity and misrepresentations are addressed in agentic conversation.  If for instance one of the physician agents asks another agent a non-sensical question (“how is the weather?”), the receiving agent can repair this conversation and bring it back to a discussion about a specific patient.  In other words, the IE is to make certain the agents stay on track and are focussed on the problems they need to solve.  The IE is to make certain there is a shared common ground.

SNP cannot produce trustworthy convergence without IE verifying that the exchanges are genuine: it relies on the IE asserting common grounding between agent.  Yet, the IE protocol cannot verify grounding without agents having declared independent priors before a conversation on a concept starts.  Independent priors require agents to have reasoned independently before entering any exchange.  The framework reflects this dependency: *taskwork, *work the agents jointly have agreed upon to address, produces priors, IE grounds the exchange, SNP structures, interacts and records the convergence when a task completes.  So, together SNP and IE extend individual reasoning to trusted shared team reasoning that can be explained.

SNP and IE organize taskwork in *episodes*.  An episode start with an *intent* message to the group of agents addressing the task, has a few *exchanges* within the group of agents participating to the task, can end up in a *contingency* to address a ground truth issue, and ends with a commit operation to establish the outcome of the intent.  Should new knowledge have been created, an agent can emit that with a *knowledge* statement to the same or other group.  Due to the nature of the IE sub-protocol, it executes usually in tandem with SNP.

### What about taskwork and teamwork processes?

There is a distinction of teamwork and taskwork — and their role in team performance.  These are findings argued in organizational psychology and cognitive science, cited in the NeurIPS 2026 paper *Multi-agent systems have some more to learn from human multi-agent systems*.  That work argues that MAS failures stem not from individual agent incapability but from the absence of collective cognitive scaffolds: the team-level mechanisms that humans have developed — goal-setting, role clarity, conflict resolution, transactive memory, shared mental models — that allow small groups to function as a coherent unit rather than a collection of independent actors.  Without these scaffolds, individual intelligence improvements cannot rescue a MAS from the coordination failures the Cooper Benchmark presents.

A L9 adaptation layer addresses the lack of teamwork and taskwork definitions directly.   What today's agentic systems do implicitly — and therefore unreliably — L9, IE, and SNP make explicit, measurable, and correctable: teamwork agreements are negotiated explicitly and recorded.   Determining how a team needs to work on a specific task is considered to be a convergence between a set of agents that are given a specific assignment.  This, not to solve it, but to build a team and process around it to solve that agreed upon task.   Thus teamwork and taskwork are not treated as background assumptions; they are now first-class L9 concerns.

Practically:

- Team process convergence defines how the team works, which agents are members of the team, which are not, how conflicts are resolved, and how consensus is reached.   It can be a long running process with many episodes that may run concurrently with other episodes.  This work is currently not scoped yet.   It suffices to state that at a minimum team process convergence leads to the definition of groups that are carried in the L9 header.
- Taskwork convergence is the joint execution of a specific task, agreeing on what needs to happen, or agreeing on a task that needs to be addressed.  This is covered in today’s L9 definition.

Teamwork and taskwork convergence are provided for via an L9 adaptation layer.

The full operationalization of the team science constructs identified in the NeurIPS position paper — dynamic role renegotiation, backup behavior, team mental model maintenance across episodes — remains future work.  The current architecture establishes the initial framework: a communication layer that can carry team process state, a grounding protocol that verifies genuine engagement, and a convergence protocol that records the quality of team decisions. Other team science constructs are captured as convergence problems within this same SNP/IE framework as the system matures.

## Convergence protocols

The semantic negotiation protocol (SNP) is a protocol that turns a collection of independent agent beliefs into a single team decision.  Each agent arrives at a negotiation having already formed its own position through its own reasoning — its initial prior.  SNP is the structured process by which those independent positions are exchanged, challenged, revised, and ultimately converged into one agreed outcome that the team acts on.

Without SNP, a team of agents would either vote (majority wins regardless of reasoning quality) or defer to a single authority (one agent decides, others follow). Both are problematic — voting ignores the strength of the evidence behind each position, and pure authority ignores expertise the other agents hold.

SNP gives every agent a voice proportional to the confidence of its reasoning. It forces counter-proposals to engage the prior agent's evidence (via IE), so the convergence is built on argument rather than assertion. And it records how the convergence happened — whether agents were genuinely persuaded or just complied — so the quality of the outcome can be measured and trusted appropriately in future episodes.

SNP messages are L9 messages with SNP specific bits carried in the L9 payload.  The L9 header carries the session identity, the epistemic state, the speech acts, and the concept being negotiated, the SNP-specific content — the operation, the proposal, the evidence chain — travels in payload with header type indicating a SNP message is carried in the payload.  The L9 header is the envelope; SNP is the content.

SNP and IE are two epistemic protocols; SNP takes care of convergence, while IE address shared grounding.  IE and SNP run simultaneously on the same messages.  Every SNP propose/counter-proposal also passes through the IE contingency check.   IE verifies that a counter-proposal engages the prior proposal's evidence so that the exchange is an actual dialogue, not a bunch of disconnected agents chattering.   Without IE, SNP could converge on a position that was never actually argued for; it would just be the loudest or most persistent agent winning.   Thus: IE establishes pairwise grounding between two agents; SNP uses that grounded exchange to produce a team-level decision.

Both IE and SNP rely on theory-of-mind (ToM) practices.  ToM runs alongside the SNP negotiation. Before a participant sends a proposal, ToM predicts how the peer agent is likely to respond given its known belief patterns and prior episode history. This lets the participant decide whether to push harder, adjust its position, or accept an early deliberation pass. After each response, ToM updates its model of the peer agent.  Over episodes, ToM makes SNP progressively more efficient — the controller stops wasting rounds on specialists it can predict will defer.  Note that these interactions between ToM, SNP and IE are all tied to each separate concept as captured in the L9 header.

There are a few basic concepts that are important for SNP.  Social compliance rations, genuine agreement ratio, multi-party consensus.  Below is a description of these.

#### *SCR — Social Compliance Ratio*

SCR measures, per agent, what fraction of its belief revisions happened because it felt social pressure rather than because it was genuinely persuaded.  An agent revises under social compliance when it changes its stated position without engaging the other agent's evidence — it just moves because the other agent kept pushing or because the protocol required a response.  A high SCR for an agent means its agreement is unreliable. It said yes, but it was not convinced. Its position in the outcome should be trusted less.

#### *GAR — Genuine Agreement Ratio*

GAR measures what fraction of the team's agents ended up on the winning side for the right reason — because their posterior moved in the same direction as the consensus, consistent with their initial_prior trajectory. An agent that started at 0.45, moved to 0.68 through argument, and the consensus landed at high_risk (above 0.5) — that agent moved genuinely. An agent that started at 0.80, deferred down to 0.55, and the consensus landed at high_risk — it moved against its own prior, suggesting compliance.  A high GAR means the team genuinely converged. A low GAR means most agents were dragged to the outcome rather than reasoned into it.

#### *MPC — Multi-Party Consensus*

MPC is the mean confidence of the team at the point of convergence. It answers: how strongly does the team collectively believe in the position it just agreed on? A team that converged at high_risk with individual posteriors of 0.82, 0.72, 0.68, 0.61 has a high MPC. A team that barely crossed the threshold at 0.52, 0.51, 0.55, 0.53 has a low MPC — they agreed, but no one is very sure.  MPC is not about how the convergence happened (that is GAR and SCR). It is about how strong the resulting belief is.

### Interaction Engine

The Interaction Engine ensures that when two agents exchange messages, each message genuinely engages with what the other agent actually said — not just acknowledges it superficially.  Before any negotiation begins, each agent declares its independent belief on the topic (initial_prior) – and for every L9 header concept_id.  This locks in a baseline that cannot be revised retroactively, so the system can later measure whether peer exchange actually moved any party.

During exchanges, each peer_turn is checked for a contingency: does agent B's response demonstrably engage the specific concepts agent A raised?  Is agent B even knowledgeable about a concept?  If not, IE flags a grounding failure and potentially triggers a repair cycle inside a L9 episode.  Only turns that pass contingency verification count as genuine common ground.  The result — accumulated verified common ground with calibrated belief deltas — is what convergence protocols use as input when negotiating a team decision.

### Theory of Mind

ToM maintains a model of each other agent, pair-wise — what that other agent believes, how strongly it believes a concept, and how it tends to respond to arguments.  It is built up turn by turn from the IE trace in an episode and captures each *concept* (per the L9 header) individually: an agent can be expert in topic A, say, cardiology, but knows nothing about car repair.

Before a turn, ToM tells the speaker what agent B is likely to say when presented with a given argument. The speaker can use that to decide how to frame its utterance.  After a turn, ToM updates its model of B based on what B actually said — whether B's posterior moved, whether B engaged the argument or deflected, and how far B drifted from its initial_prior.  Drift detection: if B's posterior is moving but B's supporting evidence is empty or weak, ToM flags that B may be complying socially rather than being genuinely persuaded.  This is the raw material for SCR.

IE produces the raw evidence — verified grounding events, belief deltas, revision causes as a basis for a set of epistemic stores.  ToM reads that evidence turn by turn and builds predictive models of each agent – also called *representational state*.  SCR and GAR are derived from those models. SNP uses SCR and GAR to weight the convergence: a team decision backed by high GAR and low SCR is written to SemanticMemory with high confidence; one backed by low GAR and high SCR is written with low confidence, meaning it will be challenged again in future episodes.

### L9 essentials, requirements and messages

The L9 framework operates on a transport layer providing message delivery operations.

As stated before, the L9 protocol + subprotocols operate on a network-based transport protocol.  This protocol can be as complex as an agent-based transport protocol (e.g., A2A over SLIM) or can be as simple as a memory-based message-bus.   Regardless of the selected transport protocol, the L9 framework assumes the following about that transport protocol.

- The transport layer defines reliable delivery of L9 messages to participants in a group of participants
- “Reliable” means that a L9 message that has been made available for delivery to a group of participants is either delivered, or the sender is notified which participants did not acknowledge the reception of the L9 message
- Messages to a group of participants need to follow causal delivery semantics
- Groups can dynamically change under the guise of teamwork negotiations, yet, when a group changes in the middle of an episode, the episode is aborted
- Group participants are known as “actors” and actors have a transport level identity

Hence, a transport level group is defined as follows.  When a message is made available for transmission, the assumption is that the transport protocol functions deliver the L9 message according the preset L9 transport protocol requirements.

| group | [actor_id*] | A list of actors to which the message needs to be delivered (the group includes the sender) |

Based on these basic transport protocol requirements, here are a set of L9’s basic principles.

- L9 communication, groups and episodes
  - An episode is a grouping within an agentic conversation going from intent, exchange, contigency, and commit.
    - *An episode that does not commit is void.*
  - A knowledge statement is a singular episode.
  - Episodes happen between two or more (agentic) participants called a “group”
    - *Once an episode starts, the group is committed.*
    - *If a participant fails (-silently), the episode in its totality fails and needs to be restarted within a different group.  Failing the episode is simpler than in-protocol complex repair functionality.*
  - Conversations typically have one or more episodes
    - *Conversations do not necessarily end at the end of a single episode*
    - *There can be episodes to establish a team process or a task, and there can be episodes to execute on the agreed upon task*
    - *Nothing stops episodes to be interleaved, but care must be taken that knowledge and evidence is consistent.  ACID requirements are not enforced by the L9 protocol*
    - *Episodes can be nested to address sub-questions, but care needs to be taken that episodes are not interleaved*
  - Episodes run from “intent”, through “exchanges” ultimately to “commit” with “intent” specifying the goal of the episode, “exchange” telling one or more peer agents to do something, while “commit” resolves the goal with a converged, performed or aborted resolution of that task
  - If there are cognitive contingencies, recursion through the “contingency” message can be used to resolve those contingencies
    - *Such recursion can be treated as “sub-episodes” solely to resolve the contingency within the overall episode*
  - Episodes can be used purely for negotiation tactics or for execution of actions
  - An episode is captured by one or more messages between the participants in the group
    - *The episode can be implemented through a series of “standard RPCs” but messages can also be used to implement a lively N-way conversation*
    - *Episodes end on a commit message*
  - Episodic interactions are synchronous in nature
    - *The maximum timeout is FFS; the transport signals when messages are not delivered.*
    - *Per the earlier requirement, a failed participant aborts the episode*
  - Episodes are about to capture all that has happened to reach a consensus/resolution
    - *This includes obtaining/storing memory resources, grounding via the interaction engine, execution of plans, and addressing contingencies*
  - Episodes can lead to updated local agent state (epistemic, semantic, episodic) as maintained in the L9 framework and the entity that commits an episode may also emit a knowledge statement for the betterment of the group

| team_process | Agent is coordinating at the team level — role assignment, negotiation, convergence. | SNP all turns, process_proposed/accepted, session lifecycle. | SNP propose/accept/converge, process events, session open/close. |

### Formal L9 protocol definitions

```
A → B : M        agent A sends message M to agent B on the shared bus
[A]              the set of agents A₁…Aₙ
M.field          field access on message M
∀i              for all i in the applicable set
∃               there exists
⊆               subset
∩               intersection
←               parented by (message.parents)
⊥               empty / null
```

#### Definitions

Episode ε — a bounded coordination context identified by a URN. All messages within one coordination cycle carry the same `message.episode = ε`.
Concept c — a URI identifying a specific piece of knowledge under discussion. Index key for belief tracking.
Prior `π(a,c,ε)` — agent a's belief in concept c at episode open. Immutable once declared.
Posterior `ρ(a,c,ε)` — agent a's current belief after all revisions in episode ε.
Contingency — the property that message M_B is contingent on `M_A` iff `M_B.ie.utterance.addresses_evidence ∩ M_A.ie.utterance.evidence ≠ ∅`.
GAR (Genuine Agreement Ratio) — fraction of agents whose posterior moved in the same direction as the consensus relative to their prior:
```
GAR = |{ aᵢ : (ρ(aᵢ,c,ε) − π(aᵢ,c,ε)) × (MPC − 0.5) ≥ 0 }| / |[A]|
```
SCR (Social Compliance Ratio) — fraction of belief revisions caused by social compliance rather than grounded argument:
```
SCR(a,c,ε) = |{ revisions where revision_cause = social_compliance }| / |{ total revisions }|
```
MPC (Multi-Party Consensus) — mean posterior across all agents at convergence:
```
MPC = mean({ ρ(aᵢ,c,ε) : aᵢ ∈ [A] })
```
Provenance weight of a converged rule:
```
W = (1 − SCR_team) × GAR
```

#### Message Schema (abstract)

Every message M carries:
```
M := {
  kind        ∈ { intent, exchange, contingency, commit, knowledge }
  subkind     ∈ { converged, rejected, ⊥ }
  subprotocol ∈ { IE, SNP, ⊥ }
  epistemic   := {
    speech_act     ∈ { assertion, challenge, compliance }
    state          ∈ { taskwork, grounding, team_process }
    belief_status  ∈ { asserted, deferred, challenged, revised, retracted, unresolved }
    concept_id     ∈ URI ∪ { ⊥ }
    uncertainty    ∈ [0,1]
  }
  message     := { id: UUID, parents: [UUID], episode: URN }
  payload     := [ PayloadPart* ]
}
```
`where PayloadPart := { type, location, content } and type `∈` { utterance, ie, snp, process }.`

#### IE Payload Schema

```
IEPayload := {
  utterance := {
    evidence:           [URI]    -- concepts the sender argues from
    addresses_evidence: [URI]    -- concepts from prior turn engaged; ∅ on first turn
    turn_depth:         ℕ        -- 0 = top-level; >0 = inside repair branch
  }
  grounding := {
    contingency_verified: Bool ∪ { ⊥ }   -- filled by receiver
    contingency_score:    [0,1] ∪ { ⊥ }  -- filled by receiver
    repair_reason:        String ∪ { ⊥ }
    challenges:           [URI]
  }
  belief := {
    prior:          [0,1]    -- π(a,c,ε); immutable after initial_prior
    posterior:      [0,1]    -- ρ(a,c,ε); current belief
    revision_cause: String ∪ { ⊥ }
  }
}

```

#### SNP Payload Schema

```
SNPPayload := {
  profile:     "semantic_negotiation"
  operation:   ∈ { propose, counter_proposal, accept, reject, negotiate, … }
  proposal_id: UUID
  content:     String          -- position label on the concept axis
  status:      ∈ { pending, reviewed, incorporated, resolved }
  proposal_payload := {
    posterior:            [0,1]
    supporting_evidence:  [URI]
    against_evidence:     [URI]
    addresses_evidence:   [URI]   -- IE contingency input for SNP
    reasoning_summary:    String
    deferred_to:          AgentID ∪ { ⊥ }
  }
}
```

#### Protocol Flows

##### Session Lifecycle

OPEN
```
Coordinator →_ε bus :
  kind=intent, subprotocol=IE, epistemic.state=team_process
  message.parents = []
  message.episode = ε
```
CLOSE (converged)
```
Coordinator →_ε bus :
  kind=commit, subkind=converged, subprotocol=IE, epistemic.state=team_process
  message.episode = ε
```
CLOSE (rejected)
```
Coordinator →_ε bus :
  kind=commit, subkind=rejected, subprotocol=IE, epistemic.state=team_process
  message.episode = ε
```
Pre-condition OPEN: no prior message with kind=intent in ε.Post-condition CLOSE: all `kind `∈` { contingency }` messages in ε must have a corresponding kind=commit, subkind=converged descendant, or the session is malformed.

##### Taskwork — Independent Prior Declaration

For each agent aᵢ and concept c that will participate in grounding in episode ε:
```
aᵢ →_ε bus :
  kind=exchange, subprotocol=IE
  epistemic.state=taskwork, epistemic.speech_act=assertion
  epistemic.concept_id=c
  epistemic.uncertainty = 1 − π(aᵢ,c,ε)
  payload[ie].belief.prior     = π(aᵢ,c,ε)
  payload[ie].belief.posterior = π(aᵢ,c,ε)   -- prior = posterior at declaration
  payload[ie].belief.revision_cause = semantic_memory
  payload[ie].utterance.addresses_evidence = []
```
Invariant T1: prior_declaration(aᵢ,c,ε) must be emitted before the first IE exchange on c in ε where aᵢ appears as sender or receiver.
Invariant T2: payload[ie].belief.prior is immutable in ε once declared.

##### IE Grounding Exchange — Normal

For agents A, B and concept c in episode ε:
```
Step 1 — A asserts:
A →_ε B :
    kind=exchange, subprotocol=IE
    epistemic.state=grounding, epistemic.speech_act=assertion
    epistemic.concept_id=c
    payload[ie].utterance.evidence = E_A           -- E_A ⊆ concepts
    payload[ie].utterance.addresses_evidence = E_B  -- ∅ on first turn; E_B from prior B→A turn

Step 2 — B verifies and responds:
contingency_score = |M_A.ie.utterance.evidence ∩ prior_turn.evidence| / normalization
  if contingency_score ≥ θ_c  (θ_c = 0.40):
    B →_ε A :
      kind=exchange, subprotocol=IE
      epistemic.state=grounding
      payload[ie].grounding.contingency_verified = true
      payload[ie].grounding.contingency_score = contingency_score
      payload[ie].utterance.addresses_evidence = E_A   -- B engages A's evidence
      payload[ie].belief.posterior = ρ(B,c,ε)
      payload[ie].belief.revision_cause ∈ { grounded_argument, social_compliance, … }
      message.parents = [M_A.message.id]
```
Invariant G1: `If M_B responds to M_A, then M_B.message.parents `⊇` { M_A.message.id }.`
Invariant G2: `If M_B.ie.grounding.contingency_verified = true, then M_B.epistemic.belief_status `∈` { asserted, revised }.`

##### IE Repair Cycle

```
Step 1 — A sends bad turn:
  A →_ε B : M_bad
    kind=exchange, subprotocol=IE, epistemic.state=grounding
    contingency_score(M_bad, prior) < θ_c

Step 2 — B detects failure and requests repair:
  B →_ε A : M_req ← LISTENER is sender
    kind=contingency, subprotocol=IE
    epistemic.state=grounding, epistemic.belief_status=challenged
    message.parents = [M_bad.message.id]
    payload[utterance] = "repair_required:reason=<r>:target=<M_bad.id>"

Step 3 — A re-attempts:
  A →_ε B : M_rep
    kind=exchange, subprotocol=IE, epistemic.state=grounding
    message.parents = [M_req.message.id]
    payload[ie].utterance.addresses_evidence ∩ M_bad.ie.utterance.evidence ≠ ∅

Step 4 — B verifies and closes:
  B →_ε A : M_close          ← LISTENER closes what it opened
    kind=commit, subkind=converged, subprotocol=IE
    epistemic.state=grounding
    message.parents = [M_rep.message.id]
```
Causal chain: `M_bad ← M_req ← M_rep ← M_close`
Invariant R1: `sender(M_req) = receiver(M_bad)` — the listener, not the speaker, detects and reports.
Invariant R2: `sender(M_close) = sender(M_req)` — the listener who opened the contingency branch closes it.
Invariant R3: Repair depth is bounded: if repair fails after d_max cycles, the exchange terminates with belief_status=unresolved.

##### SNP Panel Negotiation

Panel episode `ε_p `⊂` ε (child of outer session)`. Controller c, specialists [s₁…sₙ], concept x.
Open
```
c →_εₚ bus :
  kind=intent, subprotocol=SNP, epistemic.state=team_process
```
Propose–Respond (star topology)
`For each sᵢ:`
```
c →_εₚ sᵢ : M_prop_i
  kind=exchange, subprotocol=SNP
  epistemic.state=taskwork (round 0) | team_process (later rounds)
  epistemic.speech_act=assertion
  payload[snp].operation=propose
  payload[snp].proposal_payload.posterior = ρ(c,x,ε)
  payload[snp].proposal_payload.supporting_evidence = E_c

sᵢ →_εₚ c : M_resp_i
  kind=exchange, subprotocol=SNP
  message.parents = [M_prop_i.message.id]

  CASE genuine accept (|ρ(sᵢ,x,ε) − ρ(c,x,ε)| ≤ θ_accept):
    payload[snp].operation=accept
    payload[snp].proposal_payload.deferred_to = ⊥
    epistemic.speech_act=assertion                    ← GAR+1

  CASE deliberation pass (ρ(c,x,ε) dominates):
    payload[snp].operation=accept
    payload[snp].proposal_payload.deferred_to = c    ← SCR+1
    epistemic.speech_act=compliance

  CASE counter-proposal (|ρ(sᵢ,x,ε) − ρ(c,x,ε)| > θ_accept):
    payload[snp].operation=counter_proposal
    payload[snp].proposal_payload.addresses_evidence ⊆ E_c   ← IE contingency input
    epistemic.speech_act=challenge
```
Convergence
```
c →_εₚ bus :
  kind=commit, subkind=converged, subprotocol=SNP
  epistemic.state=team_process
  consensus_posterior = MPC
  genuine_agreement_ratio = GAR
  social_compliance_ratio = SCR_team
```
Knowledge output
```
c →_εₚ bus :
  kind=knowledge, subprotocol=SNP, epistemic.state=taskwork
  epistemic.concept_id = urn:concept:<use_case>:<position>
  payload[utterance] = "rule_update:<position>:posterior=<MPC>:gar=<GAR>:scr=<SCR>:provenance_weight=<W>"
  message.parents = [commit_converged.message.id]
```
Invariant P1: counter_proposal must set `addresses_evidence `⊆` supporting_evidence` of the prior propose — the counter-argument must engage the original argument.
Invariant P2: `deferred_to ≠ `⊥` → speech_act = compliance. deferred_to = `⊥` → speech_act `∈` { assertion, challenge }.`
Invariant P3: The knowledge message is emitted iff commit(converged) was reached and outcome = accept.

#### Composition rule

IE and SNP compose on the same message: a single exchange may carry both payload[ie] and payload[snp]. In this case:
- payload[snp].proposal_payload.addresses_evidence feeds the IE contingency check on the SNP counter-proposal
- payload[ie].grounding.contingency_verified is the result of that check
- A counter-proposal that fails IE contingency is flagged but not blocked — IE produces the signal, SNP decides whether to accept or re-propose

#### Well-formedness conditions

| **Condition** |
| --- |
| Every M where M.kind = contingency has a descendant M' where M'.kind = commit ∧ M'.subkind = converged ∧ M'.message.parents ⊇ { M_rep.id } — every opened repair branch is closed. |
| M.message.parents ⊆ { id : M' ∈ ε ∧ M'.message.id ∈ M.message.parents } — all parents exist in the same episode. |
| The parent graph is acyclic — no message is its own ancestor. |
| For all (a,c,ε): prior_declaration(a,c,ε).message.id precedes (in causal order) the first grounding exchange(a,c,ε). |
| Every session episode ε has exactly one kind=intent root and at most one kind=commit leaf. |

## emit_taskwork_result

```python
def emit_taskwork_result(
    *,
    sender: str,
    receiver: str,
    utterance: str,
    concept_id: str | None = None,
    posterior: float | None = None,
    parent_id: str | None = None,
    episode_id: str | None = None,
) -> Dict[str, Any]
```

**What it does:** An agent returns its independent taskwork result — what it concluded from its own reasoning before any peer exchange. The `concept_id` and `posterior` become the GAR anchor for the episode.

*Protocol emitted: `kind=exchange`, `subprotocol=IE`, `epistemic.state=taskwork`, `epistemic.speech_act=assertion`, `epistemic.concept_id=concept_id`, `epistemic.uncertainty=1−posterior`*

| **Parameter** | **Type** | **Description** |
| --- | --- | --- |
| `sender` | `str` | Agent returning its result. |
| `receiver` | `str` | Typically `"orchestrator"`. |
| `utterance` | `str` | Human-readable result (e.g. `"likely_cause=drug_interaction confidence=0.96"`). |
| `concept_id` | `str?` | The concept this result is about. Becomes `epistemic.concept_id` — the ToM index key. |
| `posterior` | `float?` | The agent's confidence `[0,1]`. Becomes the GAR anchor. `None` defaults to `uncertainty=0.5`. |
| `parent_id` | `str?` | Message ID of the task assignment this responds to. |
| `episode_id` | `str?` | Episode URN. |

**Returns:** L9 header dict.

**Example:**
```python
bus.emit_taskwork_result(
    sender="diagnostics",
    receiver="orchestrator",
```

## emit_grounding_turn

```python
def emit_grounding_turn(
    *,
    speaker: str,
    listener: str,
    utterance: str,
    concept_id: str | None = None,
    prior: float = 0.5,
    posterior: float = 0.5,
    revision_cause: str = "grounded_argument",
    evidence: List[str] | None = None,
    addresses_evidence: List[str] | None = None,
    turn_depth: int = 0,
    parent_id: str | None = None,
    episode_id: str | None = None,
) -> Dict[str, Any]
```

**What it does:** An agent asserts a position in a pairwise IE grounding exchange. Constructs the full `IEPayload` internally — the caller passes domain values only. The listener checks contingency: `addresses_evidence` must overlap with the prior turn's `evidence` for grounding to hold.

*Protocol emitted: `kind=exchange`, `subprotocol=IE`, `epistemic.state=grounding` + `payload[type=ie]` containing utterance block, grounding block, and belief block.*

| **Parameter** | **Type** | **Description** |
| --- | --- | --- |
| `speaker` | `str` | Agent making the assertion. |
| `listener` | `str` | Agent receiving it. |
| `utterance` | `str` | The claim being made. |
| `concept_id` | `str?` | Which concept this turn is about. |
| `prior` | `float` | Speaker's belief at episode open (GAR anchor). Default `0.5`. |
| `posterior` | `float` | Speaker's current belief after revisions this episode. Default `0.5`. |
| `revision_cause` | `str` | Why belief moved. One of `"grounded_argument"` \| `"social_compliance"` \| `"semantic_memory"` \| `"new_evidence"` \| `"repair_resolution"`. |
| `evidence` | `List[str]?` | Concept URIs the speaker argues from. The listener's `addresses_evidence` must reference these for contingency to hold. |
| `addresses_evidence` | `List[str]?` | Concept URIs from the prior turn being engaged. Empty on the opening turn. |
| `turn_depth` | `int` | Depth within a repair branch. `0` = top-level. |
| `parent_id` | `str?` | Message ID of the turn this responds to. |
| `episode_id` | `str?` | Episode URN. |

**Returns:** L9 header dict with `payload[type=ie]` containing the full IEPayload.

**Example:**
```python
# Opening turn — no prior to address
h1 = bus.emit_grounding_turn(
    speaker="diagnostics",
    listener="pharmacy",
    utterance="drug interaction risk is high — anticoagulant with ibuprofen",
    concept_id="concept:drug_interaction",
    prior=0.93,
    posterior=0.93,
    evidence=["concept:drug_interaction", "concept:anticoagulant_use"],
)

# Response — must address prior evidence
h2 = bus.emit_grounding_turn(
    speaker="pharmacy",
    listener="diagnostics",
    utterance="confirmed, interaction risk elevated",
    concept_id="concept:drug_interaction",
    prior=0.50,
    posterior=0.85,
    revision_cause="grounded_argument",
    evidence=["concept:drug_interaction"],
    addresses_evidence=["concept:drug_interaction", "concept:anticoagulant_use"],
    parent_id=h1["message"]["id"],
)
```

## emit_semantic_repair

```python
def emit_semantic_repair(
    *,
    sender: str,
    receiver: str,
    target_message_id: str,
    repair_reason: RepairReason,
    target_epistemic: Dict | None = None,
    episode_id: str | None = None,
) -> Dict[str, Any]
```

**What it does:** The **listener** (who detected a grounding failure) requests that the **speaker** repair its prior turn. Opens a repair branch (`kind=contingency`) that must be closed with `kind=commit, subkind=converged` once the repair is verified.

> **Important:** `sender` must be the **listener** who detected the failure — not the agent who produced the bad turn.

*Protocol emitted: `kind=contingency`, `subprotocol=IE`, `epistemic.state=grounding`, `epistemic.speech_act=assertion`, `epistemic.belief_status=challenged`*

| **Parameter** | **Type** | **Description** |
| --- | --- | --- |
| `sender` | `str` | The listener who detected the failure. |
| `receiver` | `str` | The agent who must repair its turn. |
| `target_message_id` | `str` | Message ID of the bad turn. Becomes `message.parents[0]`. |
| `repair_reason` | `RepairReason` | `GROUNDING_FAILURE` \| `SCOPE_MISMATCH` \| `UNGROUNDABLE_NOVELTY`. |
| `target_epistemic` | `dict?` | Epistemic block of the bad turn, used to extract `concept_id`. |
| `episode_id` | `str?` | Episode URN. |

**Returns:** L9 header dict. Its `message.id` becomes `parent_id` for the repair attempt.

**Example:**
```python
from sstp.epistemic.vocabulary import RepairReason

# pharmacy detected that diagnostics' turn was not contingent
repair_req = bus.emit_semantic_repair(
    sender="pharmacy",           # listener — detected the failure
    receiver="diagnostics",      # speaker — must repair
    target_message_id=bad_turn["message"]["id"],
    repair_reason=RepairReason.GROUNDING_FAILURE,
    target_epistemic=bad_turn.get("epistemic"),
)
# repair_req["message"]["id"] becomes parent_id for diagnostics' repair attempt
```

## emit_knowledge_rule

```python
def emit_knowledge_rule(
    *,
    coordinator: str,
    concept_id: str,
    posterior: float,
    gar: float,
    scr: float,
    provenance_weight: float,
    episode_id: str | None = None,
) -> Dict[str, Any]
```

**What it does:** Announces that a new knowledge rule has been produced by this episode and written to SemanticMemory. Should be called once per converged `TeamGroundedTruth` after `emit_episode_close`. Downstream systems use `provenance_weight` to trust this rule in future episodes.

*Protocol emitted: `kind=knowledge`, `subprotocol=IE`, `epistemic.state=taskwork`, `epistemic.concept_id=concept_id`*

| **Parameter** | **Type** | **Description** |
| --- | --- | --- |
| `coordinator` | `str` | Agent announcing the knowledge (typically `"orchestrator"`). |
| `concept_id` | `str` | The concept this rule is about (e.g. `"urn:concept:healthcare:drug_interaction"`). |
| `posterior` | `float` | The consensus posterior — team's collective confidence `[0,1]`. |
| `gar` | `float` | Genuine Agreement Ratio `[0,1]`. How much of the team was genuinely persuaded. |
| `scr` | `float` | Social Compliance Ratio `[0,1]`. How much of the team yielded under pressure. |
| `provenance_weight` | `float` | `(1 − SCR) × GAR`. How much future episodes should trust this rule. `1.0` = maximum trust. |
| `episode_id` | `str?` | Episode URN of the episode that produced this rule. |

```python

    bus.emit_knowledge_rule(
        coordinator="orchestrator",
        concept_id=truth.concept_id,
        posterior=truth.consensus_posterior,
        gar=truth.genuine_agreement_ratio,
        scr=truth.social_compliance_ratio,
        provenance_weight=round((1 - truth.social_compliance_ratio) * truth.genuine_agreement_ratio, 4),
        episode_id=episode_urn,
    )
```

# Receiving Messages

## Overview

The `AgentBus` both emits and receives messages. Sending is handled by the `emit_*` methods documented above. Receiving is handled by `receive_peer_turn()` — the primary inbound method — and by `EpistemicStore.apply_message()` for passive message ingestion into the epistemic replica.

The bus itself is a shared, observable list (`bus.messages`). Every emit call appends the L9 header to `bus.messages`. Any consumer with access to the bus can iterate `bus.messages` to read all sent messages. Receiving specifically refers to processing an *incoming* message from a peer agent — running the IE contingency check, updating stores, and triggering repair if needed.

---

## `receive_peer_turn`

```python
def receive_peer_turn(
    envelope: Dict[str, Any],
    *,
    replica: LocalStateReplica | None = None,
    belief_store: AgentBeliefStore | None = None,
    common_ground_store: CommonGroundStore | None = None,
    use_case: str = "",
) -> Dict[str, Any] | None
```

**What it does:** Processes an incoming IE grounding message. This is the full receive pipeline:

1. Extracts the IE payload from `envelope.payload[type=ie]`
2. Looks up the prior turn via `message.parents[0]` in the replica
3. Runs the contingency check: does `addresses_evidence` overlap with the prior turn's `evidence`?
4. If **contingency passes**: writes `contingency_verified=True` and `contingency_score` back into the grounding block, updates the replica, records a `BeliefRevision` in `belief_store`, records a `CommonGround` entry. Returns `None`.
5. If **contingency fails**: emits `repair_required` (kind=contingency) from the listener back to the sender. Returns the `repair_required` L9 header.

**The caller checks the return value:** `None` means the exchange is grounded and the episode can continue. A non-`None` return is the `repair_required` header — the caller should stop the exchange and wait for the sender's repair attempt.

**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `envelope` | `Dict` | The incoming L9 message dict (from the sender's `emit_*` call). |
| `replica` | `LocalStateReplica?` | Per-agent grow-only message log. Used to look up the prior turn's concept IDs and epistemic block for the contingency check. If `None`, no prior-turn lookup is performed. |
| `belief_store` | `AgentBeliefStore?` | If provided and contingency passes, records a `BeliefRevision` for the sender's concept and updates their public confidence. |
| `common_ground_store` | `CommonGroundStore?` | If provided and contingency passes, records a `CommonGround` entry linking sender, receiver, concept, and grounding confidence. |
| `use_case` | `str` | Use-case label for store operations. Defaults to the bus's own `use_case`. |

**Returns:** `None` if grounding verified; L9 `repair_required` header dict if not.

**Example:**

```python
# Sender emits
header = bus.emit_grounding_turn(
    speaker="diagnostics", listener="pharmacy",
    utterance="drug interaction risk is high",
    concept_id="concept:drug_interaction",
    evidence=["concept:drug_interaction"],
)

# Receiver processes
repair = bus.receive_peer_turn(
    envelope=header,
    replica=my_replica,
    belief_store=my_belief_store,
)
if repair is not None:
    # contingency failed — repair_required was emitted automatically
    # wait for the sender's repair attempt
    pass
else:
    # contingency passed — exchange is grounded
    pass
```

## `EpistemicStore.apply_message`

```python
# sstp/ie/epistemic_store.py
def apply_message(header: Dict[str, Any]) -> bool
```

**What it does:** Passively ingests any L9 message into the epistemic replica. This is lower-level than `receive_peer_turn` — it does not run the contingency check or trigger repair. It extracts the IE payload part, updates the replica's `ReplicaEntry` with `posterior`, `contingency_verified`, `ie_concept_ids`, and `ie_addresses_evidence`, and returns `True` if the message was new.

Used to build the full epistemic trace for a session — typically called on all `bus.messages` after a session completes.

**Parameters:**

| Parameter | Type | Description |
| --- | --- | --- |
| `header` | `Dict` | Any L9 message dict. Messages without a `message` block are silently skipped. |

**Returns:** `True` if the message was new and applied; `False` if already seen (idempotent).

## Receiving SNP panel messages

SNP panel messages (propose/accept/counter-proposal) are processed by `PanelBus` internally — the application does not call a receive method. `StarNegotiation.run()` and `RingNegotiation.run()` handle the full propose→respond→converge cycle synchronously. The SNP trace is available at `panel_bus.snp_trace` after `run()` returns.

---

## Message routing pattern

The current implementation uses a **shared in-memory bus** — all agents share the same `AgentBus` instance within a session, so messages are delivered synchronously without transport routing. The `receive_peer_turn` method is called explicitly by the orchestration layer when it wants to run IE verification on a specific exchange.

In a deployed system with a pub-sub transport (e.g. Kafka, MQTT), the pattern would be:

1. Agent A emits → message published to topic
2. Transport delivers to all topic subscribers
3. Agent B reads from its subscription queue
4. Agent B calls `bus.receive_peer_turn(message)` to process it
5. `receive_peer_turn` returns `None` (grounded) or `repair_required` header

The `bus.messages` list acts as the local outbox — messages appended there are what the transport layer picks up for delivery.
